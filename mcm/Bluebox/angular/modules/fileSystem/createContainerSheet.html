The Key Cascade tree structure manages all the object keys in its leaves. The height and node degree determine how many keys are available as well as the performance.

<- This example tree has height 2 and 2 partition bits.

New container will have height {{ newContainer.sdosHeight }} and {{ newContainer.sdosPartitionBits }} partition bits. The partitions will have {{ newContainer.sdosPartitionBits | sdosPartitionBitsToSize }} slots (2^bits).

The Cascade will support up to {{ newContainer | sdosCapacity | numberFormat }} object keys.

Tree height Partition bits Deletable key

The master key for each cascade is stored encrypted with a deletable key.

Please select where the deletable key should come from. Note that the secure-delete property is derived from this deletable key.

Dummy key
Static test key: Uses the same static key every time. Only useful for testing and debugging.
User password: Uses a password to derive a deletable key. Secure deletion is provided by setting a new password and forgetting the old one.
You must provide the current password before using the container, and provide a new password before starting secure deletion.
Trusted Platform Module (TPM): Uses TPM hardware to securely en/decrypt the master key. The deletable key is stored inside the TPM, and can never leave it. Secure deletion is provided by the TPM which can securely replace internal keys.
Batch deletion
{{ newContainer.sdosBatchDelete }}

Names of deleted objects get saved to a log so that secure deletion (re-keying) can be done later for a batch of objects.

Object contents are deleted from the object store but the keys remain, and the names remain allocated, until batch deletion is triggered. This batch delete mode can be turned on/off at any time.
